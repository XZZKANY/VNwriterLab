# 一、先定 V1 的核心数据对象

我建议 V1 先只保留 7 个核心对象，不要再多。

## 1. 项目 Project

代表一部 VN 作品。

包含：

- 项目名称
- 简介
- 类型
- 创建时间
- 最近编辑时间
- 项目设置

它是最外层容器。

------

## 2. 路线 Route

代表一条故事线。

例如：

- 共通线
- 林夏线
- 会长线
- 真结局线
- 坏结局线

包含：

- 路线名称
- 路线类型
- 排序
- 描述

它属于某个项目。

------

## 3. 场景 Scene

这是最核心对象。

一个场景就是一个可编辑、可跳转的剧情节点。

包含：

- 场景名称
- 所属路线
- 所属章节（可选）
- 状态
- 摘要
- 内容块列表
- 前置条件
- 后续连接

它属于某条路线，也属于某个项目。

------

## 4. 内容块 Block

场景内部的最小编辑单元。

类型包括：

- 旁白块
- 对白块
- 选项块
- 条件块
- 注释块

包含：

- 类型
- 顺序
- 内容
- 附加配置

它属于某个场景。

------

## 5. 角色 Character

角色资料对象。

包含：

- 名字
- 身份
- 外观
- 性格
- 目标
- 秘密
- 所属路线
- 备注

它属于某个项目。

------

## 6. 设定 Lore

设定资料对象。

分类可以是：

- 地点
- 术语
- 世界规则
- 事件

包含：

- 名称
- 分类
- 描述
- 标签
- 关联项

它属于某个项目。

------

## 7. 条件/标记 Variable / Flag

这是逻辑控制对象。

V1 可以分两类：

- 布尔标记 Flag
- 数值变量 Variable

例如：

- 已见过旧校舍事件
- 林夏好感度
- 是否进入真结局前置线

它属于某个项目，被场景和选项引用。

------

# 二、对象之间的关系

下面是最关键的关系。

## 1. 项目 与 路线

一个项目可以有多条路线。

关系：

- Project 1 → N Route

------

## 2. 项目 与 场景

场景最终也都属于某个项目。

关系：

- Project 1 → N Scene

但场景通常先挂在路线下面管理。

------

## 3. 路线 与 场景

一条路线包含多个场景。

关系：

- Route 1 → N Scene

这是剧情结构的骨架。

------

## 4. 场景 与 内容块

一个场景包含多个内容块。

关系：

- Scene 1 → N Block

这是编辑器的核心。

------

## 5. 场景 与 场景

场景之间有跳转关系。

关系：

- Scene N ↔ N Scene

这种关系通常由：

- 选项块
- 条件块
- 默认跳转

来建立。

这也是分支图页的核心数据来源。

------

## 6. 场景 与 角色

一个场景可涉及多个角色，一个角色也能出现在多个场景。

关系：

- Scene N ↔ N Character

这通常是多对多关系。

------

## 7. 场景 与 设定

一个场景可引用多个设定，一个设定可在多个场景中出现。

关系：

- Scene N ↔ N Lore

------

## 8. 场景 / 选项 / 条件 与 标记变量

场景进入条件、选项显示条件、选择结果都会读写标记变量。

关系：

- Scene N ↔ N Flag/Variable
- Block N ↔ N Flag/Variable

------

## 9. 角色 与 设定

角色可能关联某些设定。

例如：

- 某角色属于某组织
- 某角色常出现于某地点
- 某角色知道某术语背后的秘密

关系：

- Character N ↔ N Lore

这个在 V1 可以先做轻关联。

------

# 四、页面与对象的对应关系

这一步很重要，因为它能帮你知道每一页到底在操作什么。

------

## 1. 项目首页

主要读取：

- Project
- Route
- Scene
- Character
- Lore

主要展示：

- 项目概况
- 统计信息
- 最近编辑内容

主要动作：

- 新建项目内容
- 进入某页面

项目首页更偏“读”和“导航”。

------

## 2. 剧情编辑页

主要读写：

- Scene
- Block
- Flag / Variable
- Character
- Lore

主要动作：

- 编辑场景
- 新增块
- 配选项跳转
- 配条件
- 查看关联角色与设定

这是最重的“写操作页”。

------

## 3. 分支图页

主要读取：

- Route
- Scene
- Scene-to-Scene Links
- Flag / Variable（摘要）

主要动作：

- 查看结构
- 新增节点
- 新增连接
- 修改节点关系
- 跳回编辑页

这是“结构管理页”。

------

## 4. 角色页

主要读写：

- Character
- Character ↔ Scene
- Character ↔ Lore

主要动作：

- 新建角色
- 编辑角色
- 查看出场场景
- 查看关联设定

这是“人物资料页”。

------

## 5. 设定页

主要读写：

- Lore
- Lore ↔ Scene
- Lore ↔ Character

主要动作：

- 新建设定
- 编辑设定
- 查看引用位置

这是“世界观资料页”。

------

## 6. 预览页

主要读取：

- Scene
- Block
- Scene Links
- Flag / Variable

主要动作：

- 模拟流程推进
- 触发选项
- 计算条件
- 展示当前路径

这是“运行态页面”。

------

# 六、V1 最关键的数据中心

如果只看产品骨架，我会说 V1 实际上围绕 3 个中心转：

## 1. Scene

这是主核心。
 因为用户真正写的是场景。

## 2. Links between Scenes

这是 VN 性质的来源。
 因为用户真正管理的是“场景如何连接”。

## 3. Flags / Variables

这是分支逻辑的来源。
 因为用户真正做的是“条件控制”。

所以从架构角度看，你的产品不是一个普通文档系统，而是：

**Scene + Link + Logic 三层结构**

这个理解很重要。

------

# 七、V1 推荐的数据层思路

从实现角度，V1 可以尽量简单。

## 1. 项目层

保存整个作品级别信息。

## 2. 内容层

保存：

- 路线
- 场景
- 内容块

## 3. 逻辑层

保存：

- 场景连接
- 条件
- 标记
- 数值变量

## 4. 资料层

保存：

- 角色
- 设定

## 5. 运行层

保存：

- 当前预览状态
- 已触发标记
- 当前路径

------

# 八、对象优先级判断

如果按重要性排序，我会这样排：

## P0

- Project
- Scene
- Block
- Scene Link

## P1

- Route
- Flag / Variable

## P2

- Character
- Lore

这个排序不是说角色和设定不重要，
 而是从系统骨架来说，先没有 Scene / Link / Block，这产品根本站不住。

# 九、最小 IA 版本

如果你现在还想再收缩，V1 最小可用的信息架构甚至可以只有：

- Project
- Scene
- Block
- Link
- Flag
- Character
- Lore

其中 Route 甚至都可以先作为 Scene 的一个字段存在，而不是独立复杂模块。

也就是说：

- 每个场景有 route_type
- 分支图按 route_type 分组展示

这样可以更轻。

# 十、以我之见，V1 最值得提前想清楚的 3 个架构问题

## 1. 场景是不是唯一编辑核心

我的建议是：**是。**

不要让“章节”“路线”“分支点”都变成同级复杂对象。
 场景作为核心会最稳。

------

## 2. 跳转关系是存在场景里，还是独立建边

我的建议是：

V1 在编辑上可以写在选项块里，
 但数据层最好抽象成 **独立 link**。

因为分支图页会非常依赖这个。

------

## 3. 条件逻辑要不要先做得很复杂

我的建议是：**不要。**

V1 先只支持：

- 布尔标记
- 数值比较
- 简单满足/不满足

够用了。

------
