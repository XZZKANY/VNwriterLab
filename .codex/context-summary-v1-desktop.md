## 项目上下文摘要（Task 1：初始化桌面工程与基础测试工具链）

生成时间：2026-04-04  

### 1. 相似实现分析

- **实现1**：`V1 信息架构图（IA）.md`
  - 模式：以 7 个核心对象组织内容，强调项目、路线、场景、内容块、角色、设定等层级关系。
  - 可复用：V1 的对象边界定义可直接映射到后续路由与页面结构。
  - 需注意：当前阶段只做桌面壳层，不提前实现业务对象。
- **实现2**：`V1页面清单.md`
  - 模式：以 6 个核心页面为主导航，突出项目首页、剧情编辑、分支图、角色、设定、预览的固定入口。
  - 可复用：页面命名与导航顺序。
  - 需注意：不要扩展额外页面，避免偏离 V1。
- **实现3**：`导航结构 + 页面跳转关系图.md`
  - 模式：顶层导航优先按创作心智排序，首页、剧情、分支图、预览是高频入口。
  - 可复用：侧边栏导航顺序与默认首页落点。
  - 需注意：布局只需承载导航与内容区，不引入多余导航层级。

### 2. 项目约定

- **命名约定**：界面文案使用简体中文；组件与文件名采用项目既有英文标识符约定。
- **文件组织**：React 代码位于 `src/`，桌面端配置位于 `src-tauri/`，测试放在组件相邻位置或 `src/test/`。
- **导入顺序**：先第三方库，再本地模块。
- **代码风格**：保持最小实现、职责单一、避免过度抽象。

### 3. 可复用组件清单

- 暂无现成代码组件可复用；仓库当前仅有规划文档。
- 后续可复用的方向：路由壳层、全局布局、测试初始化配置。

### 4. 测试策略

- **测试框架**：Vitest + jsdom + Testing Library。
- **测试模式**：先写失败测试，再实现最小通过代码。
- **参考文件**：`src/app/layouts/AppShell.test.tsx`。
- **覆盖要求**：导航链接存在性、布局壳层渲染、首页默认内容。

### 5. 依赖和集成点

- **外部依赖**：`react-router-dom`、`zustand`、`reactflow`、`@tauri-apps/plugin-sql`、`vitest`、`jsdom`、`@testing-library/*`、`@types/node`。
- **内部依赖**：`src/main.tsx` 作为路由入口，`src/app/layouts/AppShell.tsx` 作为布局壳层。
- **集成方式**：`createBrowserRouter` + `RouterProvider` + `Outlet`。
- **配置来源**：`src-tauri/tauri.conf.json`、`package.json`、`vitest.config.ts`。

### 6. 技术选型理由

- **为什么用这个方案**：与审批通过的架构一致，React Router 适合桌面应用的壳层导航，Vitest 适合 Vite 生态。
- **优势**：路由与布局解耦，测试可以独立验证导航契约。
- **劣势和风险**：仓库起步阶段没有代码骨架，脚手架输出与计划可能存在文件名差异，需要以官方生成结构为准适配。

### 7. 关键风险点

- **并发问题**：当前任务无明显并发风险。
- **边界条件**：测试需要在 jsdom 环境下运行，否则 DOM 断言会失效。
- **性能瓶颈**：当前仅初始化壳层，不存在实际性能压力。
- **安全考虑**：本任务不涉及安全逻辑。
