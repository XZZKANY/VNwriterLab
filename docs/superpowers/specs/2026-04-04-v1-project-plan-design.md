# VN 写作工具 V1 项目规划书

## 1. 文档定位

本文档是一份面向 V1 启动阶段的项目规划书，用于把现有的产品想法、PRD、页面结构、信息架构和开发路线，整理成一份可执行、可沟通、可验收的统一方案。

这份文档的主要用途有两个：

- 作为内部执行依据，帮助后续开发、设计与功能取舍围绕同一目标推进
- 作为对外沟通材料，让协作者或潜在合作方快速理解项目方向与阶段目标

本文档以内部执行为主，但保留清晰的产品背景、价值表达和阶段规划，确保它既能指导开工，也能支撑后续介绍项目。

## 2. 项目背景与立项目的

### 2.1 项目背景

当前大量视觉小说创作者拥有故事构思、人物设计和分支叙事的创作意愿，但并不具备引擎使用经验，也不希望在创作初期被脚本语法、变量逻辑和项目工程结构打断。

这类用户通常依赖多种工具拼凑工作流：

- 用 Word 写正文
- 用 Notion 存设定
- 用 XMind 画分支
- 用 Excel 记录条件和变量

这种工作方式会带来三个突出问题：

1. 信息分散，难以统一维护
2. 分支结构复杂后容易失控
3. 技术门槛过早介入，打断创作状态

### 2.2 立项目的

本项目的立项目的是打造一款面向不会引擎的视觉小说创作者的结构化写作工具，让用户在一个统一环境中完成从故事整理到分支预览的基础创作闭环。

项目不是要在 V1 阶段构建完整游戏引擎，也不是要覆盖所有创作场景，而是优先解决一个高频而明确的问题：

`让新手作者能够写故事、管理分支、检查逻辑，而不用先学代码。`

### 2.3 产品核心价值

V1 阶段的核心价值集中在三点：

- 好写：帮助用户快速进入创作状态
- 好理清：帮助用户看懂剧情结构和分支关系
- 不用编程：让用户不依赖脚本知识也能表达基础交互逻辑

项目的核心价值主张可以浓缩为一句话：

`写故事，不写代码。`

## 3. V1 项目目标与成功定义

### 3.1 V1 项目目标

V1 的目标是打造一款本地桌面应用优先的 VN 创作工作台，使用户能够独立完成以下完整流程：

1. 创建一个 VN 项目
2. 建立角色、设定、路线与场景结构
3. 用块式编辑方式编写剧情内容
4. 配置基础分支、条件和跳转逻辑
5. 通过结构图与预览功能验证故事流程是否走通

### 3.2 V1 成功定义

V1 是否成立，不以功能数量为判断依据，而以是否能完成基础创作闭环为判断依据。

如果一个不会引擎的新手用户，能够在产品中完成一个可阅读、可分支、可预览的 VN 故事骨架，那么 V1 就是成功的。

### 3.3 V1 成功标准

从使用结果上，V1 至少应支持用户完成以下成果：

- 创建 1 个项目
- 创建至少 1 条主线或共通线
- 创建多个场景并完成基础排序
- 写入旁白、对白、选项等核心内容块
- 配置至少 1 个分支点
- 配置至少 1 个条件判断
- 建立基础角色与设定资料
- 在分支图中看懂并检查整体结构
- 在预览中完整走通至少 1 条故事路径

从产品认知上，用户应能明确感受到：

- 比 Word 更适合写 VN
- 比 Notion 更适合管理分支
- 比直接学习引擎更容易开始

## 4. V1 范围与边界

### 4.1 V1 必做范围

V1 聚焦 7 个核心模块：

1. 项目创建与结构管理
2. 剧情编辑器
3. 分支结构图
4. 角色库
5. 设定库
6. 条件 / 标记系统
7. 简单预览系统

### 4.2 V1 不做范围

为控制项目复杂度，以下内容明确不纳入 V1：

- 正式游戏导出
- 团队协作
- 云同步
- 复杂动画演出
- 美术与音频素材深度管理
- 高级 UI 编辑
- AI 大功能
- 社区、商业化与平台生态

### 4.3 范围控制原则

所有 V1 需求判断都必须围绕以下原则：

`是否直接服务于“建项目 → 写场景 → 配分支 → 看结构 → 做预览”的创作闭环。`

如果某项需求不能直接增强这条主链路，就不应优先进入 V1。

## 5. 产品形态与总体方案

### 5.1 形态定义

V1 产品形态定义为：

`本地桌面应用优先的单机创作工作台。`

### 5.2 桌面端优先原因

选择桌面应用优先，主要基于以下判断：

1. 目标用户已有明显的本地创作习惯  
   用户本身就常在本地文档工具中长期写作，桌面形态更接近其已有工作流。

2. V1 核心是持续创作体验  
   长时间编辑、频繁切换结构视图、查阅角色设定和即时预览，更适合作为桌面工作台来承载。

3. 本地优先更利于验证核心闭环  
   在早期阶段，项目应优先验证写作、结构管理与预览体验，而不是先承担账号体系、同步和多人协作的复杂度。

4. 有利于控制研发范围  
   桌面端优先意味着团队可以把精力集中在编辑器、结构图、数据关系和预览逻辑上，而不是被部署与在线能力分散资源。

### 5.3 产品定位结论

V1 不是一个轻量网页工具，也不是一个完整游戏引擎，而是一个帮助新手作者完成 VN 结构化创作的本地桌面工作台。

## 6. 用户与核心使用场景

### 6.1 目标用户

V1 的核心用户是：

- 有故事想法，想做视觉小说
- 擅长写作，但不擅长技术实现
- 需要处理角色路线、多结局和分支叙事
- 希望在一个工具中统一管理创作材料与结构

### 6.2 核心使用场景

V1 主要服务以下 5 条使用路径：

1. 开始项目  
   创建项目，选择结构模板，输入故事简介，初始化路线结构。

2. 编写剧情  
   在场景中插入旁白、对白、选项、条件和注释，完成剧情表达。

3. 整理结构  
   在分支图中查看全局关系，发现断链节点和混乱结构。

4. 查阅资料  
   在写作过程中查看角色资料、设定信息和关联场景。

5. 预览测试  
   从开头或指定节点开始推进，确认路线、条件与选项逻辑是否正确。

## 7. V1 核心功能规划

### 7.1 模块一：项目创建与结构管理

目标：让用户从故事出发，而不是从技术工程出发。

核心能力包括：

- 新建项目
- 输入项目名称与一句话简介
- 选择基础结构模板
- 创建路线 / 章节 / 场景层级
- 新增、删除、重命名、排序和移动结构节点

### 7.2 模块二：剧情编辑器

目标：让用户像搭故事一样写 VN，而不是像写代码一样处理逻辑。

核心能力包括：

- 场景编辑工作区
- 旁白块
- 对白块
- 选项块
- 条件块
- 注释块
- 内容块插入、排序、删除和编辑

### 7.3 模块三：分支结构图

目标：帮助用户从全局视角理解剧情结构与分支关系。

核心能力包括：

- 场景节点展示
- 场景之间的连线关系
- 主线 / 支线 / 结局的区分
- 条件摘要显示
- 从结构图回到对应编辑内容

### 7.4 模块四：角色库

目标：集中管理角色信息，避免角色设定散落在不同工具中。

核心能力包括：

- 角色列表与详情
- 姓名、身份、性格、外观、目标、秘密等基础资料
- 与路线和场景的关联展示

### 7.5 模块五：设定库

目标：集中管理世界观资料，降低设定冲突与引用遗漏。

核心能力包括：

- 地点管理
- 术语管理
- 世界规则管理
- 事件资料管理
- 与场景的基础关联

### 7.6 模块六：条件 / 标记系统

目标：让用户无需编程也能表达基础交互逻辑。

核心能力包括：

- 布尔标记 Flag
- 数值变量 Variable
- 选项选择后修改标记或变量
- 场景进入条件
- 选项显示条件

### 7.7 模块七：预览系统

目标：帮助用户验证剧情路径是否可读、可走通、可检查。

核心能力包括：

- 从开头预览
- 从指定节点预览
- 点击选项推进流程
- 按条件显示路径结果

## 8. 信息架构与页面结构

### 8.1 V1 页面收敛原则

为降低认知负担，V1 页面层面收敛为 6 个核心页面，不额外扩张复杂导航。

### 8.2 核心页面

1. 项目首页  
   用于项目概览、进度查看、最近编辑与快速入口。

2. 剧情编辑页  
   用于完成最核心的场景写作和逻辑配置。

3. 分支图页  
   用于查看全局结构、发现问题、调整分支关系。

4. 角色页  
   用于管理角色资料与基础关联信息。

5. 设定页  
   用于管理地点、术语、规则和事件。

6. 预览页  
   用于阅读与测试当前作品流程。

### 8.3 页面组织原则

页面设计必须服务两个目标：

- 让用户容易找到当前要做的事
- 让用户在写作、整理和验证之间切换成本尽可能低

## 9. 核心数据对象与关系

### 9.1 V1 核心对象

V1 保持 7 个核心对象即可：

- Project
- Route
- Scene
- Block
- Character
- Lore
- Variable / Flag

### 9.2 对象角色定义

Project 是最外层项目容器。  
Route 是故事结构的主骨架。  
Scene 是剧情组织的核心节点。  
Block 是最小编辑单元。  
Character 和 Lore 负责资料层支撑。  
Variable / Flag 负责逻辑驱动。

### 9.3 核心关系

V1 规划中最关键的关系包括：

- Project 1 → N Route
- Project 1 → N Scene
- Route 1 → N Scene
- Scene 1 → N Block
- Scene N ↔ N Scene
- Scene N ↔ N Character
- Scene N ↔ N Lore
- Scene / Block N ↔ N Variable / Flag

### 9.4 数据结构原则

所有页面与功能都应建立在统一核心对象之上，避免出现编辑器、结构图、角色库和预览系统使用不同逻辑模型的情况。

原则上：

`Scene 是创作主轴，Block 是编辑最小单元，Route 是结构骨架，Variable / Flag 是逻辑控制核心。`

## 10. 项目实施计划与阶段划分

### 10.1 阶段总原则

V1 推进遵循以下原则：

- 先保证主链路，再补辅助模块
- 先保证数据和逻辑稳定，再提升交互体验
- 每个阶段结束都要形成可验证成果

### 10.2 阶段一：项目基础框架与数据底座

目标：建立桌面应用骨架与核心数据模型。

本阶段重点交付：

- 桌面端应用基础框架
- 项目创建入口
- 本地项目保存机制
- 核心对象模型定义
- 基础导航和页面壳层

里程碑结果：

`用户可以创建项目，系统可以稳定组织和保存核心对象。`

### 10.3 阶段二：剧情编辑主链路

目标：让用户真正开始写作。

本阶段重点交付：

- 剧情编辑页主界面
- 场景创建、编辑、删除、排序与移动
- 旁白块、对白块、注释块
- 块的基础增删改排
- 基础保存机制

里程碑结果：

`用户可以在项目中持续写出场景内容。`

### 10.4 阶段三：分支与条件系统

目标：让产品具备 VN 的基础分支叙事能力。

本阶段重点交付：

- 选项块
- 场景跳转关系
- Flag / Variable 基础支持
- 场景进入条件
- 选项显示条件
- 分支图基础展示

里程碑结果：

`用户不仅能写内容，还能配置基础分支和逻辑条件。`

### 10.5 阶段四：资料系统与预览闭环

目标：补齐创作资料管理与流程验证能力。

本阶段重点交付：

- 角色页
- 设定页
- 基础关联信息展示
- 从起点或指定节点开始预览
- 选项推进与条件生效

里程碑结果：

`用户可以完成“建项目 → 写剧情 → 配分支 → 查资料 → 做预览”的完整闭环。`

### 10.6 阶段五：打磨、修正与验收

目标：让 V1 形成稳定、可交付的版本。

本阶段重点交付：

- 关键流程测试与修复
- 结构问题检查
- 自动保存与异常恢复基础能力
- 页面交互一致性调整
- 文档补齐与验收准备

里程碑结果：

`V1 从可演示状态进入可交付状态。`

### 10.7 核心里程碑定义

V1 规划书设置 3 个关键里程碑：

1. 里程碑 A：能创建项目并开始写  
   对应阶段一到阶段二完成。

2. 里程碑 B：能完成分支逻辑表达  
   对应阶段三完成，这是 V1 成立的关键分水岭。

3. 里程碑 C：能完成完整创作闭环  
   对应阶段四到阶段五完成，是 V1 可交付的判断线。

## 11. 风险、约束与应对

### 11.1 项目约束

当前 V1 规划基于以下现实约束：

1. 默认以单人或极小团队推进
2. 本地桌面应用优先，不先处理云端和协作
3. 只验证核心创作闭环，不追求功能最全

### 11.2 主要风险

#### 风险一：范围膨胀

项目容易在推进中被导出、协作、AI、素材管理和高级模板等需求拉偏，导致主链路长期无法闭合。

应对原则：

- 所有新增需求先判断是否直接服务 V1 闭环
- 不属于主链路的内容统一后置

#### 风险二：剧情编辑器复杂度失控

编辑器一旦同时承载块式编辑、条件逻辑、跳转配置和引用信息，复杂度会迅速上升。

应对原则：

- 优先做稳定、清晰、可维护的基础编辑能力
- 复杂交互分阶段引入

#### 风险三：分支图实现成本过高

结构图属于产品亮点，但如果过早追求复杂布局和高交互，会拖慢整体推进。

应对原则：

- V1 只保证看懂结构、识别关系、支持回到编辑
- 复杂可视化优化后置

#### 风险四：预览与编辑数据不一致

如果编辑器保存的数据结构和预览执行逻辑不统一，用户会遇到“写出来了但跑不通”的问题。

应对原则：

- 预览系统与编辑系统必须共用同一套核心数据结构
- 条件与跳转逻辑统一建模

#### 风险五：资料系统反客为主

角色库和设定库一旦做得过深，容易把产品重心带向资料管理工具，而不是创作工具。

应对原则：

- 角色与设定先做够用，不做重型知识库
- 所有资料能力都应服务剧情写作与引用

## 12. 验收标准

### 12.1 功能验收标准

当 V1 具备以下能力时，可认定已达到基础功能验收：

- 能创建并保存项目
- 能创建路线、场景和基础结构
- 能在编辑器中完成基础剧情写作
- 能配置选项、跳转和简单条件
- 能在分支图中查看结构关系
- 能管理角色与设定基础资料
- 能在预览中完整走通至少一条路径

### 12.2 用户体验验收标准

当目标用户在首次使用过程中，能够在较低学习成本下完成一次完整流程，并明确感受到产品比通用文档工具更适合 VN 创作时，可认定已达到体验验收标准。

### 12.3 项目验收结论

V1 的验收结论不建立在“功能是否全面”之上，而建立在“是否帮助新手作者真正完成一次结构化 VN 创作闭环”之上。

## 13. 后续方向说明

V1.5 与 V2 已在现有路线图中有明确方向，但不纳入本次执行范围。

后续阶段可以围绕以下方向继续推进：

- 结构问题提醒
- 更完善的模板体系
- 角色与设定的引用追踪
- 全局搜索
- 自动保存与恢复增强
- 更强的项目统计能力
- 更复杂的创作管理与差异化能力

这些内容应建立在 V1 稳定交付之后，再进入下一轮规划和实现。

## 14. 结论

VN 写作工具 V1 的核心任务不是构建一个庞大的全能平台，而是把最关键的创作主链路打通。

只要 V1 能让不会引擎的新手作者，在本地桌面环境中完成从灵感整理、剧情编写、分支配置到结构检查和预览验证的全过程，这个项目就具备继续向 V1.5 和 V2 推进的基础。

因此，V1 的项目策略应始终保持克制：

- 聚焦核心闭环
- 控制范围边界
- 优先稳定而非堆功能
- 用可验证成果推进每个阶段

这将是项目在早期阶段最重要的成功条件。
