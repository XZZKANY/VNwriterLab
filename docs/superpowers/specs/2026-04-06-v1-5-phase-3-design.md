# V1.5 第三批增强设计（已完成记录）

## 1. 目标

继续在不重做架构的前提下，把当前编辑体验从“基本可维护”提升到“更稳”。这一批不新增大模块，重点处理删除后的失效引用和结构问题可诊断性，减少编辑器里“表面看不出，但数据已经坏掉”的状态。

## 2. 范围收敛

### 2.1 变量删除闭环

在编辑页变量面板支持删除变量，并同步处理所有已存在引用：

- 条件块中的 `condition.conditions[].variableId`
- 选项块中的 `effectVariableId`
- 当前选中的 `selectedVariableId`

处理规则：

- 删除变量后，条件块里引用该变量的条件项改为 `variableId: null`
- 删除变量后，选项块里的 `effectVariableId` 改为 `null`，`effectValue` 保留原值
- 如果删除的是当前选中变量，自动切换到下一个可用变量；若无变量则置空

### 2.2 场景删除后的跳转修复

第二批已经在删场景时清理了 `links`，但 `choice.metaJson.targetSceneId` 仍可能残留悬空引用。

本批补齐：

- 删除场景后，所有指向该场景的 `choice` 块 `targetSceneId` 自动清为 `null`
- 现有 `links` 清理逻辑继续保留

### 2.3 分支图问题明细

在已有“只看问题节点”基础上，补充问题原因说明，让图页不仅知道“哪里有问题”，还知道“为什么有问题”。

问题原因收敛为：

- 非结局场景且没有任何出边
- 没有任何入边且不是起始场景
- 存在条件块引用了已删除变量或未选择变量
- 存在选项块跳转到不存在的场景
- 存在选项块副作用引用了已删除变量

约束：

- 问题明细只做展示层派生，不回写数据
- 诊断以当前 store 数据为准，兼容历史持久化中的脏数据

## 3. 方案比较

### 方案 A：删除时主动修复 + 图页被动诊断

优点：

- 用户新操作不会继续制造脏数据
- 图页还能兜底暴露历史遗留问题
- 改动面可控，符合现有架构

缺点：

- 需要同时改 store 和 graph 派生逻辑

### 方案 B：只做图页诊断，不做删除时修复

优点：

- 改动小

缺点：

- 用户每次删除都会继续留下坏引用，只是“看到了”而已

### 方案 C：新增统一数据修复层

优点：

- 理论上更系统

缺点：

- 明显超出 V1.5 范围，会演变成架构重构

结论：

采用方案 A。删除动作负责修复新增脏数据，图页负责把历史和当前残留问题显式化。

## 4. 关键设计

### 4.1 变量删除协议

删除变量动作收敛在 `useEditorStore`，因为变量当前完全属于 editor store。

动作需要同步更新：

- `variables`
- `selectedVariableId`
- `scenes[].blocks[].metaJson`

为了避免逻辑散落，应增加针对 block 元数据的小型清理函数：

- 清理条件块引用
- 清理选项副作用引用

### 4.2 场景删除时的 choice 清理

场景删除由 `projectStore` 发起，但 choice 元数据存在 editor store 里，因此应在同步 editor scenes 时一起做修复：

- 清理所有 `targetSceneId === deletedSceneId` 的 choice 元数据
- 同时保留非失效字段，如 `label`、`effectVariableId`、`effectValue`

### 4.3 图页问题明细协议

继续使用 `graphData.ts` 作为唯一图派生入口，新增问题摘要结构，例如：

- `sceneId`
- `sceneTitle`
- `issues: string[]`

GraphPage 只负责渲染，不在页面里重复做诊断。

## 5. 测试策略

### 5.1 编辑页与 store

- 删除变量后，条件块和选项副作用引用会自动清空
- 删除变量后，变量选中态正确回退
- 删除场景后，choice 元数据里的 `targetSceneId` 被清空

### 5.2 图页

- 对于无出边 / 无入边节点显示问题原因
- 对于无效变量引用和无效跳转显示问题原因
- “只看问题节点”时问题明细跟随当前子图收缩

### 5.3 持久化

- 删除变量并修复引用后，重载仍保持清理后的结果

## 6. 风险与控制

- 元数据清理要小心保持未失效字段不被误删
- 图页问题明细是派生结果，不能和修复逻辑耦死，否则会造成展示和存储相互污染
- 删除变量后条件块会变成“未选择变量”状态，这属于刻意保留的可编辑降级，而不是隐式删除整条条件
