# VN Writer Lab 前端工作台重构设计

## 目标

在不重写仓储层、状态管理架构和桌面端运行时的前提下，对 VN Writer Lab 的整体导航、首页与各主页面的前端体验进行一次工作台式重构，使产品从“页面功能集合”提升为“统一、专业、清晰、顺手的创作工具”。

本次设计重点解决两个用户痛点：

- 前端整体观感偏丑，缺乏一致性与专业感。
- 多页面切换与核心创作流程不够顺手，首页和全局导航无法清晰支持“先看概览，再继续写作”。

## 范围

- 重构整体导航结构，采用“左侧主导航 + 顶部上下文栏 + 主内容工作区”。
- 将项目首页重做为“项目工作台”，强化项目概览、最近编辑、待处理事项和继续写作入口。
- 统一全站视觉规范：色彩、留白、卡片、按钮、表单控件、标题层级、状态标签、空状态。
- 重组关键页面骨架与交互重心：
  - `项目首页`
  - `剧情编辑`
  - `分支图`
  - `多视图`
  - `角色`
  - `设定`
  - `预览`
- 保持现有路由与核心业务能力可用，在不改变主要功能入口的前提下提升体验。

## 非目标

- 不重写 repository runtime、SQLite 适配层或 Tauri 侧逻辑。
- 不进行大规模 Zustand store 重构。
- 不新增大型业务功能模块。
- 不调整核心领域模型与数据结构。
- 不在本轮引入新的 UI 框架、组件库或设计系统依赖。

## 用户目标与设计基调

### 用户目标

根据本轮确认结果，产品应优先满足以下使用目标：

1. 用户打开软件后先快速理解当前项目概况。
2. 用户可以在首页明确看到下一步最自然的动作。
3. 用户可以从首页一键回到最近创作上下文继续写作。

### 视觉与交互基调

- 视觉方向：**A + C，偏 A**
  - A：极简专业
  - C：编辑器式效率感
- 导航结构：**左侧主导航 + 顶部上下文栏**
- 体验原则：
  - 克制
  - 清晰
  - 统一
  - 低噪音
  - 高层级感
  - 高频操作一步可达

## 方案比较与选定方案

### 方案 1：工作台式重构（选定）

特点：

- 左侧固定主导航承担模块切换。
- 顶部上下文栏承担当前项目、当前页面、快速动作与搜索。
- 首页作为项目工作台，强调“看懂项目 + 继续写作”。
- 所有页面采用统一骨架，减少各页割裂感。

优点：

- 最符合“极简专业为主，保留效率感”的目标。
- 能同时优化观感与操作路径。
- 适合作为全产品统一改版基线。

### 方案 2：创作沉浸优先

特点：

- 首页弱化概览，强调直接进入正文。
- 编辑页绝对中心化，其他页面工具化。

不选原因：

- 与用户确认的“先看项目概览，再继续写作”主流程不一致。

### 方案 3：控制台式重构

特点：

- 信息密度更高，强调专业工具感和高频操作。

不选原因：

- 容易重新走向拥挤、压迫和不够清爽，与“偏 A”冲突。

## 整体信息架构

### 全局框架

采用以下统一结构：

```text
左侧主导航 + 顶部上下文栏 + 主内容工作区
```

页面布局抽象如下：

```text
┌────────侧栏导航────────┬──────────────────── 顶部上下文栏 ────────────────────┐
│ 品牌 / 当前项目入口     │ 页面标题｜当前项目｜最近场景｜全局搜索｜继续写作｜保存状态 │
│                        ├─────────────────────────────────────────────────────┤
│ 工作台                  │                                                     │
│ - 项目首页              │                主内容工作区                          │
│                        │                                                     │
│ 创作                    │                                                     │
│ - 剧情编辑              │                                                     │
│ - 分支图                │                                                     │
│ - 多视图                │                                                     │
│                        │                                                     │
│ 资料                    │                                                     │
│ - 角色                  │                                                     │
│ - 设定                  │                                                     │
│                        │                                                     │
│ 输出                    │                                                     │
│ - 预览                  │                                                     │
└────────────────────────┴─────────────────────────────────────────────────────┘
```

### 左侧主导航

左侧主导航改为分组结构，不再平铺所有页面：

- 工作台
  - 项目首页
- 创作
  - 剧情编辑
  - 分支图
  - 多视图
- 资料
  - 角色
  - 设定
- 输出
  - 预览

交互要求：

- 当前页面高亮。
- 当前分组弱高亮。
- 侧栏在桌面端保持固定，不随内容区滚动。
- 侧栏底部可放版本信息或轻量状态信息，但不能压过主导航。

### 顶部上下文栏

顶部上下文栏固定承载以下信息与动作：

- 当前页面标题
- 当前项目名
- 最近编辑场景或当前上下文
- 全局搜索入口
- 主动作按钮
- 自动保存/状态反馈

主动作按钮按页面变化，但结构保持统一：

- 项目首页：继续写作
- 剧情编辑：新建场景 / 返回首页
- 角色页：新增角色
- 设定页：新建设定

目标不是让每页拥有不同顶部，而是让用户在所有页面都感受到相同的导航语法。

## 统一视觉规范与交互风格

### 视觉关键词

- 克制
- 清晰
- 稳定
- 低噪音
- 高信息层级
- 弱装饰、强结构

### 色彩策略

保留深色工作台方向，但从“旧式深色 UI”升级为“专业深色创作工具”：

- 背景：深而不死黑
- 面板：比背景亮一级
- 边框：弱化存在感，避免每个模块都像独立盒子
- 强调色：全站仅保留一个主强调色
- 状态色：只用于真正重要的状态区分

### 布局与留白策略

- 统一页面横向分区。
- 减少碎片化卡片数量，合并同层级信息。
- 所有页面使用统一的 spacing 节奏。
- 页面标题、说明、操作区和正文区采用固定层级节奏。

### 组件统一规范

统一以下组件表现：

- 按钮
- 输入框
- 下拉框
- 文本域
- 卡片容器
- 页面标题
- 区块标题
- 状态标签
- 列表项
- 空状态提示

规则：

- 每页只有一个最强主按钮。
- 次按钮克制，不与主按钮抢注意力。
- 危险操作降权。
- 空状态必须给出下一步动作，而不是仅显示“暂无数据”。

### 信息层级规范

全站统一为四层：

1. 页面目标
2. 当前最重要动作
3. 主要工作内容
4. 辅助信息

所有页面必须保证主内容区始终最突出，统计、导出、提示等辅助信息不得压过正文工作区。

## 页面级设计

## 项目首页：项目工作台

### 页面定位

首页不再是简单的信息展示页，而是项目工作台，目标是：

1. 快速看懂当前项目状态
2. 快速决定下一步去哪里
3. 一键继续上次创作

### 页面结构

建议采用“上轻下重”的两段式结构：

```text
[顶部总览区]
- 项目名称 / 简介
- 当前进度摘要
- 继续写作（主按钮）
- 打开分支图 / 从头预览（次按钮）

[下方工作台区]
左侧主栏：最近编辑、待处理事项、项目结构概览
右侧侧栏：项目统计、角色/设定/变量概览、导出入口
```

### 首页核心模块

#### 顶部总览区

- 项目名称
- 一句话简介
- 当前模板或项目类型
- 主按钮：继续写作
- 次按钮：打开分支图 / 从头预览

#### 最近编辑区

- 最近编辑场景标题
- 所属路线
- 当前状态
- 一句摘要
- 快捷动作：
  - 继续写作
  - 打开分支图定位
  - 从该场景预览（若合理）

#### 项目结构概览区

- 路线数量
- 场景总数
- 开场场景
- 结局场景数
- 路线列表与每条路线的场景数

建议表现方式：

- 上方 4 个指标卡
- 下方路线概览列表

#### 待处理事项区

首页需明确指示当前项目的下一步工作：

- 待补充场景数
- 待检查逻辑场景数
- 未完成场景数
- 结构问题提示

#### 右侧辅助侧栏

- 项目统计
- 角色数量 / 设定数量 / 变量数量
- 导出入口
- 搜索入口

约束：导出与统计不可抢占“继续写作”的主视觉位置。

## 剧情编辑页：主创作工作区

### 页面定位

剧情编辑页是最高频页面，必须从“功能堆叠页”改为“主创作工作区”。

### 页面结构

建议采用三栏式结构：

```text
[顶部上下文栏：当前项目 / 当前场景 / 新建场景 / 返回首页 / 打开分支图]

[左栏] 场景树 / 路线结构
[中栏] 当前场景编辑主区
[右栏] 场景属性 / 变量 / 当前块相关配置
```

### 三栏职责

#### 左栏：结构导航

- 路线分组
- 场景树
- 当前场景高亮
- 快速新建场景
- 快速切换场景

#### 中栏：正文与块编辑

- 场景标题、摘要、状态
- 块列表
- 新增块按钮组
- 块内容编辑区

#### 右栏：上下文配置

- 当前场景附加信息
- 变量编辑入口
- 条件块 / 注释块 / 选择块配置
- 与当前编辑动作相关的辅助信息

### 关键约束

- 中栏最宽，保持创作主区绝对中心。
- 高频操作始终保持在可见区。
- 变量与配置不抢正文主视觉。
- 选中块后，右栏跟随上下文变化，减少用户在正文区上下跳转。

## 分支图页：结构分析工作区

### 页面定位

分支图页不是单纯“看图”，而是帮助用户看懂结构、发现问题并快速跳回修正。

### 页面结构

```text
[顶部上下文栏：当前项目 / 当前筛选 / 回到最近场景 / 继续写作]

[左侧可折叠面板] 筛选 / 问题分类 / 图例
[中间主区] 分支图画布
[右侧详情面板] 当前节点信息 + 快捷动作
```

### 结构重点

#### 左侧面板

- 问题分类
- 结构异常
- 筛选条件
- 图例说明

#### 中间画布

- 节点和边是主角
- 起始场景、结局场景、异常节点使用有限标签和状态色区分
- 聚焦、缩放、定位等操作清晰可见

#### 右侧详情面板

- 场景标题
- 所属路线
- 当前状态
- 前后链接摘要
- 快捷动作：
  - 去编辑
  - 设为当前场景
  - 从该节点预览

## 多视图页：结构观察台

### 页面定位

多视图页不承担重编辑职责，而是提供不同角度的结构观察和跳转入口。

### 页面结构

```text
[顶部上下文栏]

[视图切换区：大纲 / 状态 / 路线]
[主内容区：当前视图]
[右侧小面板：当前项摘要 + 快捷跳转]
```

### 设计要求

- 每个视图只回答一个问题，避免信息过载。
- 点击条目后可快速跳转到编辑页或图谱页。
- 不做成复杂 BI 面板。

## 角色页与设定页：统一资料管理界面

### 共用交互模型

角色页和设定页统一采用同一套结构：

```text
[顶部上下文栏]

[左栏] 列表 / 搜索 / 新增
[中栏] 详情编辑
[右栏] 辅助信息 / 关联信息 / 编写提示
```

目标是用户学会一个页面后，另一个页面零成本迁移。

### 角色页

左栏：

- 角色列表
- 搜索
- 新增角色

中栏：

- 基础信息
- 身份 / 外观 / 性格
- 目标 / 秘密
- 备注

右栏：

- 所属路线
- 使用提示
- 创作辅助信息

### 设定页

左栏：

- 设定列表
- 分类筛选
- 新建设定

中栏：

- 名称
- 分类
- 描述
- 标签

右栏：

- 分类说明
- 标签辅助信息
- 使用建议

## 预览页：成果检查页

### 页面定位

预览页应更接近作品阅读体验，而不是继续维持后台编辑界面感。

### 页面结构

```text
[顶部上下文栏：当前项目 / 当前起点 / 回到编辑 / 回到首页]

[中间主区] 预览内容
[顶部或右上轻量工具] 重新开始 / 跳转起点 / 简要状态
```

### 设计约束

- 视觉更安静。
- 减少编辑器噪音。
- 强调成果验证而非配置操作。

## 实施边界与顺序

### 实施边界

本轮优先处理前端体验重构，不处理深层架构重写：

- 纳入：导航、视觉、骨架、页面交互组织、组件统一
- 不纳入：repository 重写、状态架构重写、领域模型迁移、新功能扩张

### 建议实施顺序

#### 阶段 1：壳层与设计系统

- 左侧导航分组化
- 顶部上下文栏
- 全局 spacing / 色彩 / 卡片 / 表单 / 按钮统一
- 页面标题区统一

#### 阶段 2：首页工作台

- 项目总览区
- 最近编辑区
- 待处理事项区
- 项目结构概览区
- 统计 / 导出侧栏

#### 阶段 3：高频页面重构

- 剧情编辑页
- 分支图页
- 多视图页

#### 阶段 4：资料与输出页面统一

- 角色页
- 设定页
- 预览页

## 风险与控制

### 风险 1：页面更好看但更难用

控制方式：

- 每页仅保留一个主动作。
- 所有设计判断都回到“继续写作主流程”验证。

### 风险 2：各页面风格重新分裂

控制方式：

- 先定义壳层与视觉规范，再改业务页。
- 同类页面必须采用同构交互模型。

### 风险 3：编辑页改造影响高频操作效率

控制方式：

- 场景切换、新建场景、块编辑、变量调整都必须保留短路径。
- 不把高频操作藏进深层面板或二级菜单。

### 风险 4：视觉升级后信息密度失衡

控制方式：

- 极简不等于空。
- 效率感不等于拥挤。
- 所有页面都按“主内容 > 主动作 > 辅助信息”排序。

## 验收标准

### 视觉层

- 全站页面风格统一。
- 左侧导航、顶部栏、主内容区形成稳定框架。
- 颜色、间距、卡片、按钮、表单控件统一。
- 页面不再显得拼接、碎片、模块割裂。

### 可用性层

- 用户打开软件后，能快速理解当前项目与下一步动作。
- 从首页进入继续写作的路径更短、更明确。
- 编辑页切换场景、写块、调配置更顺手。
- 分支图、角色、设定等页面更容易理解和操作。

### 一致性层

- 所有页面属于同一产品体系。
- 同类页面使用同类布局与交互。
- 次级信息不会压过主流程。

### 回归层

- 现有核心页面功能不丢失。
- 主要导航与关键交互保持可用。
- 改版后的本地测试仍可验证主要页面功能。

## 成功定义

若本次重构成功，用户应明显感受到：

- 产品终于像一个成熟的创作工具，而不是一组拼接页面。
- 首页知道该干嘛，并且“继续写作”路径清晰。
- 编辑页更顺手，结构与配置不再互相打架。
- 其他页面终于像同一套系统中的不同工作台，而不是分散工具页。
