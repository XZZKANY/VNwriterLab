# V1.5 第二批增强设计（已完成记录）

## 1. 目标

在不重做现有架构的前提下，继续补齐 V1 到 V1.5 之间最影响日常创作效率的缺口，让用户可以更完整地维护场景信息、清理无效节点，并在分支图里更快定位结构问题。

本批只扩展 `editor`、`projects`、`graph` 三条既有主链，不新增 `repository/service` 层，不进入拖拽、复杂图编辑、导出、协作或 AI。

## 2. 范围收敛

### 2.1 场景基础信息编辑

在编辑页为当前场景补上最小基础信息表单，支持：

- 标题
- 摘要
- 场景类型
- 状态
- 是否起始场景
- 是否结局场景

约束：

- 场景信息仍以 store 为唯一真实来源
- 起始场景在同一项目内最多只能有一个
- 继续复用当前受控组件模式，不引入局部草稿系统

### 2.2 场景删除闭环

允许用户从场景树删除场景，并保持项目结构、编辑器状态和跳转关系一致。

最小闭环包括：

- 从 `useProjectStore.currentProject.scenes` 删除
- 从 `useEditorStore.scenes` 删除
- 清理所有指向该场景的 `links`
- 清理该场景内 `choice` 块产生的 `links`
- 如果删除的是当前选中场景，自动切换到下一个可用场景；若不存在则置空
- 删除后重排同路线剩余场景的 `sortOrder`，并重算 `isStartScene`

### 2.3 分支图筛选生效

让当前图页已有的筛选入口真正作用到图数据，而不是停留在静态 UI。

本批实现：

- 路线筛选：支持查看全部路线或单条路线
- 只看问题节点：仅显示疑似需要处理的场景

问题节点定义收敛为：

- 非结局场景且没有任何出边
- 没有任何入边且也不是起始场景

约束：

- 过滤只影响图页展示，不修改底层数据
- 条件摘要与连线摘要跟随当前筛选结果收缩

## 3. 方案比较

### 方案 A：全部逻辑仍收敛在既有 store 与纯函数内

优点：

- 延续当前仓库模式，改动面可控
- 测试容易接入既有页面级与纯函数级验证
- 不会引入新的同步层

缺点：

- `useProjectStore` 与 `useEditorStore` 的职责边界继续偏重，需要谨慎同步

### 方案 B：引入新的场景管理中间层统一调度

优点：

- 理论上可以把场景结构与编辑信息同步集中处理

缺点：

- 明显超出当前仓库风格
- 会把 V1.5 做成局部架构重构，风险高

### 方案 C：把所有新能力先做在页面本地状态里

优点：

- 看起来快

缺点：

- 与当前 persist 架构冲突
- 状态源会分裂，预览与图页容易失真

结论：

采用方案 A。继续使用 store 作为唯一数据源，图页筛选放在页面局部状态，图数据过滤继续收敛在 `graph` 纯函数内。

## 4. 关键设计

### 4.1 场景信息更新协议

在 `useEditorStore` 内新增场景级更新 action，用于按 `sceneId` 局部更新场景元数据。

同时在 `useProjectStore` 中新增对应同步 action，确保项目结构里的场景元信息与编辑器一致。

最稳妥的实现方式是：

- `useProjectStore` 负责项目场景元信息更新
- `useEditorStore` 负责当前编辑场景和块数据更新
- 页面触发场景元信息编辑时，主入口调用项目 store，再同步到 editor store

这样可以避免只改 editor 导致项目首页和图页看到旧数据。

### 4.2 场景删除协议

场景删除应由 `useProjectStore` 发起，因为路线内顺序和起始场景属于项目结构语义。

删除时项目 store 负责：

- 移除项目场景
- 重排同路线场景
- 修正起始场景标记

然后 editor store 同步：

- 移除对应 `scenes`
- 清理相关 `links`
- 修正 `selectedSceneId`

### 4.3 图页筛选协议

保留 `buildSceneGraph` 作为基础图数据生成器，再在 `graph` 模块新增轻量过滤函数。

理由：

- 原始图数据构造与页面筛选语义分离，更容易测试
- 不需要在 React 页面里堆过多条件分支

## 5. 测试策略

### 5.1 编辑页

- 新增场景信息编辑回归测试
- 新增删除场景后结构同步、选中切换、连线清理测试

### 5.2 项目 / store

- 补项目结构层删除与起始场景重算验证
- 持久化层验证场景元信息编辑和删除后的恢复结果

### 5.3 图页

- 新增路线筛选测试
- 新增问题节点筛选测试
- 验证条件摘要和连线摘要随筛选变化

## 6. 风险与控制

- 双 store 同步仍是本批主要风险，因此所有场景元信息与删除逻辑都必须由测试覆盖 editor/project 两侧结果。
- 问题节点定义故意保持最小，只覆盖“无入边 / 无出边”两类明显结构问题，不引入复杂拓扑分析。
- 起始场景切换规则必须保持确定性，避免删除后出现多个起始场景或没有任何可进入起始场景。
