# V1.5 下一阶段设计

## 1. 目标

在当前主链已经可用的前提下，继续解决“已经能写，但治理与继续创作入口还不够顺”的问题。本阶段不再重复建设编辑、预览、分支图基础能力，而是围绕结构治理、项目首页快速入口和模板增强做收口。

本阶段只扩展 `graph`、`projects` 两条主链，并继续消费既有 `editor` / `preview` 数据，不新增 `repository/service` 层。

## 2. 范围收敛

### 2.1 结构治理增强

在现有分支图问题明细基础上，补齐更贴近创作质量的提醒：

- 空场景提醒
- 问题分类显示
- 项目首页统计继续复用问题总数

约束：

- 诊断继续收敛在 `graphData.ts` 纯函数层
- 不新增自动修复器
- 不把问题规则散落到页面里

### 2.2 项目首页快速继续创作

在项目首页增加最小继续创作入口：

- 最近编辑场景摘要
- 继续写作按钮
- 打开分支图按钮
- 从头预览按钮

约束：

- 入口只基于当前 store 派生，不新增“最近记录”独立存储层
- 若缺少最近编辑信息，允许安全降级到当前选中场景或首个场景

### 2.3 项目模板增强

为项目创建补最小模板选择：

- 线性短篇模板
- 多结局模板
- 共通线 + 角色线模板

约束：

- 模板只初始化项目结构，不预填正文
- 仍由 `useProjectStore` 负责创建后的真实结构数据
- 第一版不做模板市场、远程模板或复杂模板编辑器

## 3. 方案比较

### 方案 A：继续沿用 store + 纯函数派生

优点：

- 与当前仓库风格一致
- 变更面最小
- 测试接入最直接

缺点：

- 页面与派生函数的边界需要继续保持克制

### 方案 B：新增项目仪表盘层统一调度首页、模板和问题规则

优点：

- 看起来更“完整”

缺点：

- 明显超出当前代码风格
- 会让文档和实现同时滑向局部架构重构

结论：

采用方案 A。结构问题继续收敛在 `graph/lib`，项目首页入口与模板创建继续收敛在 `projects` feature 内。

## 4. 关键设计

### 4.1 结构治理协议

在 `graphData.ts` 现有 `issueSummaries` 基础上扩展问题分类，例如：

- `emptyScene`
- `noIncoming`
- `noOutgoing`
- `invalidVariable`
- `invalidTargetScene`

页面只负责展示分类结果，不负责自行判定。

### 4.2 最近编辑与继续写作协议

第一版不新增持久化字段，而是复用现有状态：

- 优先读取当前选中场景
- 若没有选中场景，则退回项目内第一个可用场景
- 项目首页按钮负责路由跳转，不在页面中复制编辑器业务逻辑

### 4.3 模板初始化协议

项目创建仍由 `useProjectStore.createProject` 作为入口，但增加最小模板参数。

模板应只决定：

- 初始路线列表
- 初始场景骨架
- 起始场景标记

模板不应负责：

- 编辑器块内容预填
- 变量、角色、设定批量初始化
- 复杂条件结构

## 7. 风险与控制

- 空场景定义如果过于激进，容易把“作者暂存中的草稿场景”误判为问题，因此第一版应只提示、不阻断
- 最近编辑若没有统一来源，不应伪造复杂历史；先做最小可靠入口
- 模板创建若一次放太多字段，会迅速越界到 V2，必须限制为结构骨架初始化
