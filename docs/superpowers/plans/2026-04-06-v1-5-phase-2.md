# V1.5 第二批增强 Implementation Plan（已完成记录）

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 补齐场景元信息编辑、场景删除闭环与分支图筛选生效三项能力，让 V1.5 继续从“能用”向“顺手”推进。

**Architecture:** 继续沿用现有 feature 内聚 + Zustand persist + 纯函数图数据 / 预览模式。第二批只扩展 `editor`、`projects`、`graph` 三条主链，不新增 repository/service 层，不进入拖拽、复杂图编辑、导出、协作或 AI。

**Tech Stack:** React 19、TypeScript、Zustand、Vitest、Testing Library

---

## 范围

### 1. 场景基础信息编辑

- 编辑当前场景的标题、摘要、场景类型、状态
- 切换是否起始场景、是否结局场景
- 保持项目结构与编辑器状态同步

### 2. 场景删除闭环

- 从场景树删除场景
- 删除后清理相关连线
- 删除后自动重排同路线顺序并修正起始场景
- 处理当前选中场景被删除后的回退

### 3. 分支图筛选生效

- 路线筛选真正生效
- 只看问题节点真正生效
- 条件摘要与连线摘要跟随筛选结果收缩

## 范围外

- 不做拖拽
- 不做复杂图布局
- 不做 OR / 嵌套条件
- 不做导出、协作、AI

## 执行顺序

1. 先做场景基础信息编辑
2. 再做场景删除闭环
3. 最后做分支图筛选生效

## 验证策略

- 每条子任务先补失败测试，再补实现
- 定向验证 editor / graph / persistence
- 最后统一执行：
  - `npm.cmd test`
  - `npm.cmd run build`

## 风险

- 场景元信息在 `project` / `editor` 双 store 间同步时容易漂移，必须通过测试锁住
- 删除场景会触发结构、连线、选中态三个面同时变化，容易漏清理
- 图页筛选若直接写在页面里容易失控，因此尽量把筛选语义收敛在 `graph` 纯函数内
