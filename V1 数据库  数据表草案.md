 **V1 数据库 / 数据表草案**

# 一、设计原则

V1 的表结构建议遵守 4 条原则：

## 1. 以 Scene 为核心

因为用户真正写的是“场景”，不是抽象章节。

## 2. Link 独立出来

因为分支图、本场景跳转、条件跳转都会依赖它。
 所以“场景之间的连接”最好单独存。

## 3. Block 作为最小编辑单元

这样剧情编辑器才能支持块式写作。

## 4. 条件先简单

V1 先支持：

- 布尔标记
- 数值变量
- 简单比较
- 满足 / 不满足 两类行为

------

# 二、V1 核心数据表总览

我建议 V1 先有这 10 张主表：

1. `projects`
2. `routes`
3. `scenes`
4. `scene_blocks`
5. `scene_links`
6. `characters`
7. `lore_entries`
8. `project_variables`
9. `conditions`
10. `preview_sessions`（可选，偏运行态）

如果想更轻，V1 甚至可以先不拆 `conditions`，直接挂在 block / link 上。
 但为了后面扩展，我还是建议先有。

------

# 三、数据表草案

------

## 1. projects

代表一个 VN 项目。

### 字段建议

- `id`
- `name`
- `summary`
- `project_type`
   例如：linear / multi_ending / route_based
- `status`
   例如：draft / active / archived
- `created_at`
- `updated_at`

### 说明

这是最外层容器。
 一个项目下面会有路线、场景、角色、设定、变量。

------

## 2. routes

代表一条路线。

### 字段建议

- `id`
- `project_id`
- `name`
- `route_type`
   例如：common / character / true_end / bad_end / hidden
- `description`
- `sort_order`
- `created_at`
- `updated_at`

### 说明

用来组织场景。
 例如：

- 共通线
- 林夏线
- 会长线
- 真结局线

### 关系

- 一个 `project`
- 对应多个 `scenes`

------

## 3. scenes

代表一个场景节点。

这是最核心表。

### 字段建议

- `id`
- `project_id`
- `route_id`
- `title`
- `slug` 或内部编号
- `summary`
- `scene_type`
   例如：normal / branch / ending
- `status`
   例如：draft / completed / needs_revision
- `chapter_label`
   V1 可选，只做文本字段
- `sort_order`
- `is_start_scene`
- `is_ending_scene`
- `notes`
- `created_at`
- `updated_at`

### 说明

每个场景是一个可编辑节点。

### 为什么要有 `scene_type`

方便分支图里快速区分：

- 普通场景
- 分支点
- 结局

### 为什么 `chapter_label` 先做文本

因为 V1 不一定要单独建 `chapters` 表，先轻量一点。

------

## 4. scene_blocks

场景里的内容块。

### 字段建议

- `id`
- `scene_id`
- `block_type`
   例如：narration / dialogue / choice / condition / note
- `sort_order`
- `character_id`
   对白块时可用
- `content_text`
- `meta_json`
- `created_at`
- `updated_at`

### 说明

这是剧情编辑器真正编辑的内容。

------

### `block_type` 的建议含义

#### narration

旁白块

- 用 `content_text`

#### dialogue

对白块

- `character_id`
- `content_text`

#### choice

选项块

- 选项文案可放 `content_text`
- 去向、效果可放 `meta_json`
- 或走独立 `scene_links`

#### condition

条件块

- 条件描述 / 条件引用可存在 `meta_json`

#### note

注释块

- 用于作者备注

------

### 为什么要有 `meta_json`

因为不同 block 的字段不一样。
 V1 不必过度拆表，用一个扩展字段会更灵活。

例如 choice block 的 `meta_json` 可存：

- 选项列表
- 默认跳转
- 是否允许隐藏

------

## 5. scene_links

场景之间的连接关系。

这张表非常关键。

### 字段建议

- `id`
- `project_id`
- `from_scene_id`
- `to_scene_id`
- `link_type`
   例如：default / choice / conditional / fallback
- `source_block_id`
   来源于哪个 block，通常是选项块或条件块
- `label`
- `condition_id`
   可为空
- `priority_order`
- `created_at`
- `updated_at`

### 说明

一条 link 就是一条“从哪个场景到哪个场景”的边。

------

### 为什么要独立这张表

因为它同时服务：

- 剧情编辑页跳转设置
- 分支图画线
- 预览页推进逻辑
- 结构检查

------

### 常见例子

#### 普通默认跳转

A 场景结束后 → B 场景

#### 选项跳转

“跟她去天台” → 天台场景

#### 条件跳转

若林夏好感 ≥ 3 → 特殊剧情
 否则 → 普通剧情

------

## 6. characters

角色资料表。

### 字段建议

- `id`
- `project_id`
- `name`
- `display_name`
- `role_type`
   例如：protagonist / heroine / side / antagonist
- `appearance`
- `personality`
- `goal`
- `secret`
- `relationship_to_protagonist`
- `route_id`
   可为空，一个角色也可能不只属于一条路线
- `notes`
- `created_at`
- `updated_at`

### 说明

用于角色页面和剧情编辑页侧边栏。

------

## 7. lore_entries

设定资料表。

### 字段建议

- `id`
- `project_id`
- `title`
- `lore_type`
   例如：location / term / rule / event / organization
- `description`
- `tags_json`
- `notes`
- `created_at`
- `updated_at`

### 说明

统一放：

- 地点
- 术语
- 世界规则
- 事件
- 组织

------

## 8. project_variables

项目变量表。

V1 可统一存标记和数值变量。

### 字段建议

- `id`
- `project_id`
- `name`
- `display_name`
- `variable_type`
   例如：boolean / number
- `default_value`
- `description`
- `created_at`
- `updated_at`

### 说明

例如：

- `met_mysterious_girl`
- `affection_linxia`
- `found_key`

------

### 为什么不拆成 flags 和 variables 两张表

V1 没必要。
 统一成一张更轻，靠 `variable_type` 区分即可。

------

## 9. conditions

条件表。

这是逻辑层的基础。

### 字段建议

- `id`
- `project_id`
- `name`
- `condition_type`
   例如：variable_compare / flag_check
- `variable_id`
- `operator`
   例如：= / != / > / >= / < / <=
- `compare_value`
- `description`
- `created_at`
- `updated_at`

### 说明

V1 先只支持单条件，不支持复杂嵌套。

------

### 例子

#### 布尔条件

- variable: `met_mysterious_girl`
- operator: `=`
- compare_value: `true`

#### 数值条件

- variable: `affection_linxia`
- operator: `>=`
- compare_value: `3`

------

## 10. preview_sessions（可选）

预览运行态。

如果你要支持“从当前场景开始预览”“恢复这次测试”，可以有这张表。

### 字段建议

- `id`
- `project_id`
- `start_scene_id`
- `current_scene_id`
- `state_json`
- `path_json`
- `created_at`
- `updated_at`

### 说明

V1 甚至可以先不落库，只放前端内存。
 如果要保存预览过程，再建这张表。

------

# 四、关系表建议

为了支持“关联场景”“角色出场记录”“设定引用”，最好再有几张轻量关联表。

------

## 11. scene_characters

场景与角色的多对多关系。

### 字段建议

- `id`
- `scene_id`
- `character_id`
- `relation_type`
   例如：speaker / present / mentioned

### 用途

- 某角色出现过哪些场景
- 某场景涉及哪些角色

------

## 12. scene_lore_refs

场景与设定的多对多关系。

### 字段建议

- `id`
- `scene_id`
- `lore_entry_id`
- `relation_type`
   例如：mentioned / primary_setting / hidden_clue

### 用途

- 某设定在哪些场景被引用
- 某场景涉及哪些设定

------

## 13. character_lore_refs

角色与设定的多对多关系。

### 字段建议

- `id`
- `character_id`
- `lore_entry_id`
- `relation_type`
   例如：belongs_to / knows / lives_in / related_to

### 用途

- 角色关联组织、地点、术语

------

# 五、最小可行版本的简化方案

如果你现在想更快落地，我建议用这个“轻量版”。

## 必要表

- `projects`
- `routes`
- `scenes`
- `scene_blocks`
- `scene_links`
- `characters`
- `lore_entries`
- `project_variables`

## 可延后

- `conditions`
- `scene_characters`
- `scene_lore_refs`
- `character_lore_refs`
- `preview_sessions`

### 简化做法

V1 初期可以先把条件写进：

- `scene_links.condition_json`
- 或 `scene_blocks.meta_json`

等产品跑顺了，再抽象成独立 `conditions` 表。

这会更快。

------

# 六、推荐字段风格

为了后面前后端好配合，建议统一这几个习惯：

## 通用字段

每张表都尽量有：

- `id`
- `created_at`
- `updated_at`

## 排序字段

凡是有顺序的地方都加：

- `sort_order`

比如：

- routes
- scenes
- scene_blocks

## 可扩展字段

在不确定后面会怎么扩的时候，适当加：

- `meta_json`
- `tags_json`
- `state_json`

这样 V1 不容易被表结构绑死。

------

# 七、推荐的对象层级

你可以把整个数据结构理解成：

```
Project
├── Routes
│   └── Scenes
│       ├── Scene Blocks
│       └── Scene Links
├── Characters
├── Lore Entries
└── Project Variables
```

如果进一步加关联：

```
Scene
├── Blocks
├── Outgoing Links
├── Incoming Links
├── Related Characters
└── Related Lore
```

------

# 八、最关键的 3 张表

如果只看产品生命线，我会说最关键的是：

## 1. scenes

因为这是创作核心。

## 2. scene_blocks

因为这是编辑器核心。

## 3. scene_links

因为这是 VN 的本质核心。

没有这三张表，产品就不成立。

------

# 九、推荐实现顺序

如果你现在是要从技术上开工，我建议按这个顺序建：

## 第一步

- `projects`
- `routes`
- `scenes`

先把项目结构跑起来。

## 第二步

- `scene_blocks`

先把剧情编辑器跑起来。

## 第三步

- `scene_links`

再把分支图和跳转跑起来。

## 第四步

- `project_variables`
- `conditions`

再把基础逻辑跑起来。

## 第五步

- `characters`
- `lore_entries`
- 关联表

最后补资料管理。

------

# 十、以我之见，V1 最稳的技术策略

我最建议的是：

**数据层先“半结构化”，不要一开始就过度规范化。**

也就是：

- 核心对象结构化
- 复杂配置先放 `json`
- 等真实使用一轮后，再拆得更细

因为你现在还在探索产品形态。
 如果一开始建得太死，后面很容易改痛苦。

------

# 十一、给你的一个直接结论

如果你现在只想拿一版能开工的草案，我会建议你采用这个版本：

## 核心表

- projects
- routes
- scenes
- scene_blocks
- scene_links
- characters
- lore_entries
- project_variables

## 扩展表

- conditions
- scene_characters
- scene_lore_refs
- character_lore_refs
- preview_sessions

这个组合最平衡。
