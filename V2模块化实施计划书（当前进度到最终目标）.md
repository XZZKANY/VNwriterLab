## 1. 计划目的

在当前 V1.5 已收口基线之上，以“模块 + 功能清单”的方式推进到 V2 最终目标，确保：

- 路线清晰（先做什么、后做什么）
- 每个模块可独立验收
- 模块间依赖可追踪
- 全过程可本地验证、可回滚

## 2. 当前进度基线（事实）

基于现有文档与记录，当前已完成：

1. 结构治理增强（空场景提醒 + 问题分类）
2. 项目模板增强（三类模板）
3. 首页快速继续创作入口
4. 主工作区测试与构建基线可通过（`npm.cmd test`、`npm.cmd run build`）

> 说明：以上作为本计划的起点，不再重复建设。

---

## 3. V2 最终目标（交付边界）

V2 以“形成差异化能力”为目标，最终需完整交付以下四类能力：

- 能力1：伏笔追踪系统
- 能力2：高级条件系统
- 能力3：多视图创作管理
- 能力4：导出能力（可迁移成果）

最终验收定义：

- 四类能力均可演示、可测试、可本地复现
- 模块间数据语义一致（编辑器、图、预览、导出一致）
- 全量测试与构建稳定通过

---

## 4. 实施约束与原则

### 4.1 架构约束

- 延续现有架构：`feature` 内聚 + Zustand `persist` + 派生函数
- 优先复用现有模块，不引入额外复杂分层
- 同一份业务状态只能有一个权威来源，禁止视图各自维护独立真相

### 4.2 质量约束

- 每个模块必须给出：功能清单、输入输出、依赖、验收标准、回滚策略
- 每个模块完成后必须执行本地验证命令并记录
- 未满足验收门禁的模块不得进入下一模块

|---|---|---|---|

## 6. 模块实施细则

### M0：基线收口与契约统一

#### 模块目标

把“当前已完成能力”与“V2新增能力”边界一次性固化，防止后续实现方向漂移。

#### 功能清单

- 建立 V2 统一数据契约草案（伏笔对象、条件组对象、导出对象）
- 固化“已完成清单 / 待实现清单 / 非目标清单”
- 统一模块验收模板（每模块必须留痕）

#### 输入 / 输出

- 输入：现有路线文档、现有测试基线
- 输出：统一契约文档、基线收口清单、模块验收模板

#### 关键改动点

- `docs/superpowers/plans/`（路线与边界）
- `.codex/operations-log.md`、`.codex/verification-report.md`（执行与验证留痕）

#### 验收标准

- 基线文档中明确：当前状态、V2目标、非目标
- 统一契约文档可被 M1~M4 直接引用
- 本地验证通过：`npm.cmd test`、`npm.cmd run build`

#### 回滚策略

- 若契约评审未通过：回退到上一版文档，不进入 M1/M2

### M1：伏笔追踪系统

#### 模块目标

建立“伏笔 -> 回收点 -> 未回收告警 -> 路线级追踪”的创作闭环。

#### 功能清单（P0）

- 伏笔标记：在场景或块上标记伏笔
- 回收点标记：标记对应回收位置
- 关系绑定：伏笔可关联一个或多个回收点
- 未回收列表：支持按项目/路线筛选

#### 功能清单（P1）

- 路线回收进度统计（已回收/未回收）
- 按角色或设定标签查看伏笔分布

#### 输入 / 输出

- 输入：场景、块、路线、角色/设定关联数据
- 输出：伏笔实体、回收关系、未回收报告

#### 关键改动点

- `src/lib/domain`：伏笔与回收关系模型
- `src/features/editor`：标注入口
- `src/features/graph`：未回收可视提示或列表
- `src/features/projects`：汇总视图入口

#### 实施门禁

- 先完成模型与最小读写，再接 UI 展示
- 不允许先做复杂统计后补模型

#### 验收标准

- 可以创建、编辑、删除伏笔与回收点
- 可以查看未回收列表且筛选有效
- 对应单元测试与页面测试通过

#### 回滚策略

- 关系模型不稳定时，仅保留“单向伏笔标记”，回收关系功能暂时关闭

---

### M2：高级条件系统（编辑器/图/预览一致）

#### 模块目标

把当前基础条件扩展为可组合、可嵌套、可验证的条件语义，并保证执行一致。

#### 功能清单（P0）

- 条件组：支持 AND / OR
- 嵌套条件组：组内再包含条件组
- 条件校验：非法变量、非法操作符、空组拦截
- 语义预览：编辑器显示逻辑摘要

#### 功能清单（P1）

- 真结局前置条件检查模板
- 常用条件模板快速插入

#### 输入 / 输出

- 输入：变量状态、标记状态、条件表达式
- 输出：布尔结果、拦截原因、执行轨迹摘要

#### 关键改动点

- `src/features/editor`：条件编辑 UI 与输入校验
- `src/features/preview`：执行器语义对齐
- `src/features/graph`：条件摘要展示一致化
- `src/lib/domain`：条件表达结构定义

#### 实施门禁

- 先定义条件语义契约与测试，再修改 UI
- 未通过契约测试前禁止扩展页面交互

#### 验收标准

- AND/OR/嵌套在编辑器可配置
- 预览执行结果与编辑器语义一致
- 边界测试覆盖：空组、嵌套深度、非法引用

#### 回滚策略

- 若组合语义导致主链不稳定，保留“单层条件组”，临时禁用嵌套入口

---

### M3：多视图能力（同一数据源）

#### 模块目标

在不复制状态的前提下，提供节点图之外的创作视图，提升中长篇管理效率。

#### 功能清单（P0）

- 统一视图适配层（单一数据源）
- 新增大纲视图（优先）
- 视图间跳转一致（从任一视图可定位到编辑器场景）
- 复用路线筛选、问题筛选

#### 功能清单（P1）

- 卡片视图
- 时间线视图

#### 输入 / 输出

- 输入：场景结构、跳转关系、状态标签、条件摘要
- 输出：各视图渲染模型（大纲/卡片/时间线）

#### 关键改动点

- `src/features/graph`：抽离可复用派生逻辑
- `src/features/*`：新增视图页面模块
- `src/features/editor`：跨视图跳转落点

#### 实施门禁

- 先落地统一适配层，再开发第二视图
- 禁止每个视图单独维护一份派生逻辑

#### 验收标准

- 至少 1 个非图视图稳定可用（建议大纲视图）
- 各视图与编辑器定位一致，无“跳错场景”
- 大项目下交互无明显卡顿（以主观可用性 + 测试门禁判定）

#### 回滚策略

- 新视图异常时，不影响节点图主路径；可关闭新视图入口保留主链

---

### M4：导出能力

#### 模块目标

提供“可迁移创作成果”的导出能力，而非完整打包发行。

#### 功能清单（P0）

- 导出结构化 JSON
- 导出纯文本稿
- 导出引擎草稿脚本（文本脚本）

#### 功能清单（P1）

- 导出选项（是否带注释、是否带条件摘要）
- 导出一致性校验提示（导出前检查）

#### 输入 / 输出

- 输入：项目结构、场景内容、分支关系、条件、伏笔信息
- 输出：三类导出文件（JSON/TXT/脚本）

#### 关键改动点

- `src/lib`：导出转换逻辑
- `src/features/projects`：导出入口与参数界面
- `src/test`：导出快照与边界测试

#### 实施门禁

- 先实现 JSON 导出，再扩展 TXT 和脚本导出
- 每新增一种导出格式必须补充快照测试

#### 验收标准

- 三种导出均可执行并生成文件
- 导出内容与项目数据一致
- 异常输入可控失败并给出中文提示

#### 回滚策略

- 保底保留 JSON 导出；其他格式可按开关关闭

---

### M5：性能与稳定性收敛

#### 模块目标

在 V2 功能齐备后，确保中大型项目可持续使用。

#### 功能清单

- 派生计算热点盘点（图、视图、预览、导出）
- 关键路径缓存或增量计算策略
- 大项目回归用例（场景数、分支数、条件复杂度）
- 恢复与持久化一致性回归

#### 输入 / 输出

- 输入：全量 V2 功能链路
- 输出：性能与稳定性整改清单、回归测试结果

#### 关键改动点

- `src/features/graph` / `src/features/preview` / `src/features/projects`
- `src/lib`（派生与导出转换层）
- 测试集（性能相关回归样本）

#### 验收标准

- 关键交互无明显阻塞
- 回归测试通过且无新增高优先级故障

#### 回滚策略

- 若优化引入不稳定，优先回退到稳定算法并保留问题清单，不阻断主功能

#### 输入 / 输出

- 输入：各模块的交付物、测试结果、风险清单
- 输出：V2 发布包文档、最终验收报告、回滚手册

#### 关键改动点

- `docs/superpowers/plans/`（最终路线状态）
- `.codex/verification-report.md`（评分与结论）
- `.codex/operations-log.md`（执行留痕）

#### 验收标准

- `npm.cmd test` 通过
- `npm.cmd run build` 通过

#### 回滚策略

## 8. 风险矩阵与处置

| 风险 | 触发信号 | 影响 | 处置策略 |
|---|---|---|---|
| 条件语义不一致 | 编辑器与预览结果不一致 | 主链逻辑错误 | 先冻结新语义，回退到稳定语义层 |
| 多视图状态分叉 | 视图与图数据不一致 | 数据可信度下降 | 强制统一适配层，禁止视图私有状态真相 |
| 导出与内部数据偏差 | 导出缺字段/错字段 | 成果不可迁移 | 增加导出契约测试与快照基线 |
| 规模化性能退化 | 大项目交互卡顿 | 可用性下降 | 热点排查 + 增量优化 + 回退开关 |

## 9. 模块完成定义（DoD）

一个模块被判定“完成”，必须同时满足：

- 功能清单中的 P0 项全部完成
- 输入/输出与契约文档一致
- 本地验证门禁全部通过
- 留痕文档更新完成
- 对下游模块的依赖接口稳定可用

---

## 10. 立即执行顺序（按模块，不按周）

1. 启动 M0（基线收口与契约统一）
2. 并行启动 M1 与 M2（能力核心）
