导航结构 + 页面跳转关系图

# 一、导航结构

我建议 V1 用一个很清楚的顶层导航，不要复杂。

## 顶部主导航

- **项目首页**
- **剧情**
- **分支图**
- **角色**
- **设定**
- **预览**

这 6 个就够了。

------

# 二、导航设计原则

## 1. 按创作心智排，不按技术模块排

用户脑子里的顺序通常是：

- 我在写什么
- 结构怎么样
- 角色是谁
- 设定有没有冲突
- 现在能不能跑

所以导航顺序也应该尽量贴近这个逻辑。

------

## 2. “剧情”永远是中心入口

对你的产品来说，最高频页面一定是 **剧情编辑页**。
 所以：

- 登录项目后要很容易回到剧情页
- “继续写作”默认跳剧情页
- 其他页面都要能快速回剧情页

------

## 3. “分支图”和“预览”要离剧情很近

因为用户会一直在这三页之间来回切：

- 剧情页写
- 分支图页看
- 预览页试

所以这三个标签最好挨着放。

------

# 三、推荐导航顺序

## 方案 A：创作优先型

- 项目首页
- 剧情
- 分支图
- 预览
- 角色
- 设定

这个版本更适合真实使用频率。
 我更推荐这个。

因为对用户来说：

- 角色和设定是支撑创作的
- 剧情、分支图、预览才是主循环

------

## 方案 B：资料优先型

- 项目首页
- 剧情
- 分支图
- 角色
- 设定
- 预览

这个更整齐，但没那么贴近实际工作流。

------

# 四、我建议你采用的导航结构

我建议先用：

**项目首页 ｜ 剧情 ｜ 分支图 ｜ 预览 ｜ 角色 ｜ 设定**

理由很简单：

**剧情—分支图—预览**
 这三页是产品的发动机。

------

# 五、页面跳转关系图

下面用文字版把关系串起来。

## 1. 项目首页

可以跳到：

- 剧情编辑页
- 分支图页
- 预览页
- 角色页
- 设定页

### 常用按钮

- 继续写作 → 剧情页
- 新建场景 → 剧情页
- 打开分支图 → 分支图页
- 从头预览 → 预览页
- 新建角色 → 角色页
- 新建设定 → 设定页

------

## 2. 剧情编辑页

可以跳到：

- 分支图页
- 预览页
- 角色页
- 设定页

### 常用按钮

- 从这里预览 → 预览页
- 查看整体结构 → 分支图页
- 查看角色详情 → 角色页
- 查看设定详情 → 设定页

### 高频回路

**剧情页 → 分支图页 → 剧情页**
 **剧情页 → 预览页 → 剧情页**

这是最重要的两条回路。

------

## 3. 分支图页

可以跳到：

- 剧情编辑页
- 预览页

### 常用按钮

- 打开编辑 → 剧情页
- 从此处预览 → 预览页
- 新建节点后编辑 → 剧情页

### 高频回路

**分支图页 → 剧情页 → 分支图页**
 因为作者常常会：

- 在图上发现问题
- 回去改
- 再回来确认

------

## 4. 预览页

可以跳到：

- 剧情编辑页
- 分支图页

### 常用按钮

- 返回编辑 → 剧情页
- 查看当前节点结构 → 分支图页
- 重新开始 → 当前预览页内刷新

### 高频回路

**预览页 → 剧情页 → 再预览**
 这是验证闭环。

------

## 5. 角色页

可以跳到：

- 剧情编辑页
- 相关场景
- 设定页

### 常用按钮

- 插入到当前场景 → 剧情页
- 查看出场场景 → 跳对应场景的剧情页
- 查看相关设定 → 设定页

------

## 6. 设定页

可以跳到：

- 剧情编辑页
- 相关场景
- 角色页

### 常用按钮

- 插入到当前场景备注 → 剧情页
- 查看关联场景 → 跳剧情页
- 查看关联角色 → 角色页

------

# 六、核心跳转闭环

真正最重要的，不是所有页面都互跳，而是这几个闭环要顺。

## 闭环 1：主创作闭环

**剧情 → 分支图 → 剧情**

用途：

- 写到一半看整体
- 看完整体再回来补内容

这是第一核心闭环。

------

## 闭环 2：验证闭环

**剧情 → 预览 → 剧情**

用途：

- 写完一段马上试
- 发现问题立即回来改

这是第二核心闭环。

------

## 闭环 3：资料查阅闭环

**剧情 → 角色 / 设定 → 剧情**

用途：

- 写对白时查角色
- 写设定时查规则
- 查完马上回来继续写

这是第三核心闭环。

------

你可以先把它理解成这样：

```
 │         └─> 剧情编辑页
 ├─> 分支图页
 ├─> 预览页
 ├─> 角色页
 └─> 设定页
```

# 八、页面跳转优先级

## P0：必须非常顺

- 项目首页 → 剧情
- 剧情 → 分支图
- 剧情 → 预览
- 分支图 → 剧情
- 预览 → 剧情

## P1：应该顺

- 剧情 → 角色
- 剧情 → 设定
- 角色 → 剧情
- 设定 → 剧情

## P2：可以后补

- 分支图 → 角色
- 分支图 → 设定
- 预览 → 角色
- 预览 → 设定

V1 不需要全互通，先把高频链路做好。

------

# 九、全局操作建议

为了让这些跳转更自然，我建议加两个全局能力：

## 1. 返回上次编辑

用户不管去了角色页还是设定页，都能一键回到刚才在写的场景。

按钮文案可以直接叫：

- **返回刚才的场景**
- 或 **回到写作**

这个会非常实用。

------

## 2. 快速预览

无论在剧情页还是分支图页，都能很快点：

- **从这里预览**

这样验证成本会很低。

------

# 十、以我之见，V1 导航最该避免的问题

## 1. 页面过多

不要一开始拆出太多二级页面，不然产品会显得碎。

## 2. 跳转太绕

比如要先去项目首页，再去预览，这种就太慢。

## 3. 资料页和创作页断开

如果角色页、设定页只是“孤立资料库”，那价值会打折。
 一定要能很顺地回到写作现场。

------

# 十一、我给你的结论

如果用一句话概括这套导航逻辑，就是：

**以剧情页为中心，分支图页和预览页构成主工作闭环，角色页和设定页构成辅助闭环。**

这个结构很适合你现在的产品定位。
